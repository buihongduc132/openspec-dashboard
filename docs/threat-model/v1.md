# Threat Model v1 — OpenSpec Dashboard

- **Version:** 1.0
- **Date:** 2026-06-22
- **Author / Maintainer:** OpenSpec Dashboard core team
- **Scope:** Phase 0 + Phase 1 internet- and filesystem-facing surfaces (req 08 §8.10, NFR-11).
- **Governing spec:** `openspec/changes/phase0-foundations/specs/threat-model-v1/spec.md`
- **Review cadence:** reviewed at the Phase 0 milestone gate and re-reviewed at every phase that adds a new surface.

> **Status legend.** Each surface is tagged **[IN CODE]** (implemented in Phase 0,
> mitigation maps to a concrete artifact) or **[FORWARD-LOOKING]** (designed here,
> implementation deferred to Phase 3b — explicitly NOT yet mitigated in running code).

---

## Method

For every surface this document records, per req 08 §8.10 AC (a):

1. **Assets** — what is being protected.
2. **Adversaries** — who might attack and why.
3. **Attack tree** — concrete attack vectors.
4. **Mitigations** — controls that prevent or limit each vector, mapped to a
   spec requirement / task / code path. Forward-looking surfaces cite the
   future spec, not a current code path.
5. **Residual risk** — what remains after mitigation.

Phase 0 only registers **local** repository paths; remote sandboxed cloning is
designed here but its execution is gated on this threat model's sign-off
(see `phase0-foundations` design.md Non-Goals).

## 1. Project registration — path traversal  `[IN CODE]`

- **Assets:** the dashboard process's filesystem read boundary; the host
  filesystem outside the registered project root.
- **Adversaries:** an authenticated operator (Phase 0: single trusted user;
  later: any workspace member) who submits an arbitrary `rootPath`.
- **Attack tree:**
  1. Submit `rootPath = "/etc"` or `rootPath = "../../../"`.
  2. Projection layer reads and indexes files outside the intended project.
  3. Sensitive host files (SSH keys, `.env`) surface in the dashboard or audit log.
- **Mitigations:**
  - The filesystem-projection layer enforces a **path allowlist** rooted at the
    registered project path; any resolved path that escapes the root via `..`
    or symlink is rejected (`phase0-foundations` → `filesystem-projection` spec,
    requirement "Filesystem projection with atomic writes"; task 1.8).
  - Phase 0 registers **local paths only**; the remote-clone code path that
    would widen the attack surface is not reachable until the sandbox design
    below is signed off.
- **Residual risk:** a symlink inside an allowed project that points outside
  the root. Mitigation: projection resolves realpaths and rejects escapes
  (tracked in `filesystem-projection` spec).

## 2. Audit hash-chain — tampering & deletion  `[IN CODE]`

- **Assets:** the per-project append-only audit log (the source of truth per
  D-Audit / D0-3); INV-4 restorability; the verifiability guarantee (NFR-10).
- **Adversaries:** an operator or process with write access to the sidecar who
  edits, deletes, or reorders an already-appended chain entry to hide privilege
  misuse or corrupt history.
- **Attack tree:**
  1. Adversary edits a byte in an old chain entry's body or recomputes its hash.
  2. Adversary deletes a tail entry to erase an incriminating mutation.
  3. A partial write (crash mid-append) leaves a truncated / corrupt chain file.
- **Mitigations (all shipped in Phase 0):**
  - SHA-256 hash chaining from a genesis entry: every entry's hash commits to
    the previous one, so any in-place edit or reordering breaks the chain
    (`phase0-foundations` → `audit-chain` spec, requirement "Append-only hash
    chain"; task 5.2; NFR-10).
  - A tamper-detecting **chain verifier** (scheduled job) walks the chain and
    flags tampered or deleted entries (`audit-chain` spec, requirement "Chain
    verifier"; task 5.6).
  - On a verified chain break the system enters **read-only quarantine**:
    mutations return 503 while reads stay 200, and an operator incident is
    raised (`audit-chain` spec, requirement "Read-only quarantine"; tasks
    5.7 / 5.8).
  - Startup recovery handles missing / partial-write / unreadable audit files
    by re-initializing from genesis or quarantining on unreadable IO
    (`src/lib/audit/recovery.ts`; task 5.10).
- **Residual risk:** a collision in SHA-256. Mitigation: computationally
  infeasible; the verifier + quarantine make *detection* the trust anchor, not
  prevention of the edit itself.

## 3. Sandboxed clone — clone RCE  `[FORWARD-LOOKING — implementation in Phase 3b]`

> **Not yet implemented.** This section designs the remote-clone surface so the
> Phase 3b implementation has a concrete contract. It is NOT a mitigated risk
> today; Phase 0 never executes a remote clone.

- **Assets:** the dashboard host; CI/CD secrets; developer machines that clone
  the dashboard's sandboxes.
- **Adversaries:** a malicious upstream repository that ships hostile git hooks,
  submodules, or post-checkout scripts.
- **Attack tree:**
  1. Dashboard clones `https://attacker.example/repo.git`.
  2. Upstream `.git/hooks/post-checkout` runs arbitrary code on the dashboard host.
  3. Attacker exfiltrates the Postgres credentials or pivots to other projects.
- **Mitigations (to be enforced when execution lands in Phase 3b):**
  - Clone with `core.hooksPath=/dev/null` (project-workspace spec, "Sandboxed
    clone (M-7 hardened)").
  - Submodule recursion uses `--filter=blob:none` and no automatic checkout of
    untrusted branches.
  - Sandboxes live under `openspec/.dashboard/sandboxes/<uuid>/` and are the
    *only* path the projection layer reads.
- **Residual risk:** a future git vulnerability bypassing `hooksPath`. Mitigation:
  pin git version; run clones in a confined user namespace (Phase 3b task).

## 4. Agent write API — path-confinement & privilege escalation  `[FORWARD-LOOKING — implementation in Phase 3b]`

> **Not yet implemented.** No agent write API endpoint exists in Phase 0/1.

- **Assets:** the canonical `openspec/` tree; per-section ETag invariants
  (INV-7); tenant boundaries.
- **Adversaries:** a compromised or buggy agent credential that attempts to
  write outside its change, or to escalate privileges.
- **Attack tree:**
  1. Agent writes to `openspec/specs/<other-domain>/` outside its assigned change.
  2. Agent forges an `If-Match` to defeat the ETag concurrency control.
  3. Agent impersonates another tenant's change scope.
- **Mitigations (design contract for Phase 3b):**
  - Writes are confined to the change's delta paths; the path allowlist from §1
    is reused (`filesystem-projection` spec).
  - Every write requires a valid `If-Match` per INV-7 (section-ETag spec); a
    missing/stale ETag yields 409/412, never a blind overwrite.
  - Every write appends to the audit hash-chain (NFR-10) so privilege misuse is
    detectable after the fact.
- **Residual risk:** token theft between issuance and revocation. Mitigation:
  short-lived tokens + audit-chain replay detection (Phase 3a auth work).

## 5. Inbound webhooks — forgery & replay  `[FORWARD-LOOKING — implementation in Phase 3b]`

> **Not yet implemented.** No inbound webhook receiver exists in Phase 0/1.

- **Assets:** the change-archive trigger; task reconciliation.
- **Adversaries:** an attacker who forges a webhook from a fake git provider, or
  replays a captured legitimate webhook.
- **Attack tree:**
  1. Attacker POSTs a forged "push" webhook to archive an unrelated change.
  2. Attacker replays a captured webhook to double-trigger reconciliation.
- **Mitigations (design contract for Phase 3b):**
  - HMAC signature verification with a per-provider secret stored via the
    secret-hygiene layer (gitleaks-gated, never logged).
  - Replay protection via a nonce/timestamp window recorded in the audit chain.
  - Default-deny: only explicitly-registered provider secrets are accepted.
- **Residual risk:** secret rotation gap. Mitigation: documented rotation runbook
  (Phase 3b operations task).

## 6. Outbound webhooks — SSRF  `[FORWARD-LOOKING — implementation in Phase 3b]`

> **Not yet implemented.** No outbound webhook delivery exists in Phase 0/1.

- **Assets:** the dashboard's ability to make outbound network requests; internal
  services on the private network (metadata endpoints, internal APIs).
- **Adversaries:** a user who registers an outbound webhook target pointing at an
  internal address.
- **Attack tree:**
  1. User registers webhook URL `http://169.254.169.254/latest/meta-data/`.
  2. Dashboard fetches it and leaks cloud metadata / internal service responses
     back to the attacker via response body or timing.
- **Mitigations (design contract for Phase 3b):**
  - **SSRF guard / default-deny egress**: outbound webhook targets must resolve
    to public IPs; private/loopback/link-local ranges are rejected
    (integration-sync capability; req 08).
  - No response body is echoed to the requester; only a delivery status is
    recorded in the audit chain.
- **Residual risk:** DNS rebinding between resolution and connection. Mitigation:
  pin the resolved IP for the connection lifetime (Phase 3b task).

## 7. Multi-tenant data isolation — cross-tenant read/write  `[FORWARD-LOOKING — implementation in Phase 3a]`

> **Not yet implemented.** Phase 0/1 is single-user, single-project; no tenant
> boundary is enforced because none is reachable.

- **Assets:** per-workspace data; per-project audit chains.
- **Adversaries:** a user of workspace A attempting to read or write workspace B's
  specs, changes, tasks, or audit log.
- **Attack tree:**
  1. User queries `/api/projects/<uuid-of-other-tenant>` directly.
  2. User cross-references a leaked UUID to read another tenant's audit chain.
- **Mitigations (design contract for Phase 3a):**
  - Every read/write is scoped by a workspace membership check enforced in
    middleware (auth-multitenancy capability, req 09 §9.10).
  - The permission matrix is documented here and enforced in middleware
    (req 09 §9.10 AC (a)).
  - Audit chains are per-project (D-Audit) so erasure of one tenant cannot
    corrupt another's chain (NFR-10).
- **Residual risk:** IDOR via guessed UUIDs. Mitigation: UUIDs are
  cryptographically random; authorization is enforced regardless of ID
  guessability (Phase 3a RBAC task).

## 8. Public-repo publication — secret leak  `[IN CODE / PARTIAL]`

- **Assets:** the public GitHub repository
  `https://github.com/buihongduc132/openspec-dashboard`; any credentials,
  tokens, or PII that would leak if pushed.
- **Adversaries:** anyone reading the public history after an accidental commit
  of a secret.
- **Attack tree:**
  1. Developer commits `.env`, a DB URL, or an API key.
  2. Push to the public repo exposes it permanently in git history.
- **Mitigations:**
  - `.gitignore` pre-ignores `.env*`, `*.key`, `*.pem`, `secrets/`, `auth.json`,
    sidecar paths (task 1.4, done at commit `39cb79b`).
  - gitleaks pre-commit + pre-push hooks + a CI gitleaks gate are the Phase 0.6
    secret-hygiene deliverable (`secret-hygiene-gate` spec; tasks 1.12/1.13).
  - A one-time history scan of already-pushed refs (`e8a516f`, `39cb79b`) closes
    the retroactive gap (task 1.13).
  - Publication itself is gated by a two-person review + secret-scan check
    (req 09 §9.8; task 7.4).
- **Residual risk:** a secret format gitleaks does not recognize. Mitigation:
  custom gitleaks rules + the two-person review gate.

## 9. Auth — token theft & session fixation  `[FORWARD-LOOKING — implementation in Phase 3a]`

> **Not yet implemented.** No auth subsystem exists in Phase 0/1.

- **Assets:** session cookies; API tokens.
- **Adversaries:** an attacker who steals a token via XSS, network sniffing, or
  fixes a victim's session ID.
- **Attack tree:**
  1. XSS exfiltrates a non-`HttpOnly` token.
  2. Attacker fixes a session ID before login and reuses it post-login.
- **Mitigations (design contract for Phase 3a):**
  - Better-Auth (req 09.1–09.4, 09.7) with `HttpOnly`, `Secure`, `SameSite`
    cookies; session IDs rotated at privilege change.
  - API tokens are argon2id-hashed at rest; leak detection + rotation path
    (req 09.5, 09.10; Phase 3b task 6.1).
- **Residual risk:** phishing of the operator. Mitigation: WebAuthn/2FA
  (Phase 3a stretch).
